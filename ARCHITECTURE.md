# ToolHub Architecture Overview

This document provides a clear overview of the architecture for both the backend and frontend of the ToolHub application.

## High-Level Architecture
ToolHub is a modern web application consisting of a decoupled frontend and backend that communicate via a RESTful API.

### Architectural Patterns
- **Backend**: Service-Oriented **MVC (Model-View-Controller)**. 
  - **Model**: SQLModel schema definitions (`app/models`).
  - **View**: Serialized JSON responses (Pydantic schemas).
  - **Controller**: FastAPI routers (`app/api`).
  - **Services**: Business logic extracted from controllers for reusability (`app/services`).
- **Frontend**: Component-Based Architecture (Conceptually similar to **MVVM / Model-View-ViewModel**).
  - **Model**: Server state managed by React Query and local state managed by Zustand/Context.
  - **View**: React Components rendered via Tailwind CSS and Radix UI.
  - **ViewModel**: Custom React Hooks and React Query hooks that bind the data models to the UI views.

---

## 🖥 Backend Architecture

The backend is built using **FastAPI**, emphasizing high performance, easy API generation, and asynchronous execution. 

### Technology Stack
- **Framework**: FastAPI with Uvicorn server
- **Database ORM**: SQLModel (combines SQLAlchemy for DB interaction and Pydantic for data validation)
- **Migrations**: Alembic
- **Background Tasks**: `apscheduler` (used for periodic cleanup of temporary files; Celery was removed for a leaner deployment)
- **Rate Limiting**: `slowapi`
- **Authentication**: JWT-based (using `python-jose` and `passlib` for password hashing)

### Project Structure (`backend/app`)
- **`main.py`**: The application entry point. It configures the FastAPI instance, CORS, rate limiting, security headers, and includes the main API router. It also manages the application lifespan (startup/shutdown events like DB init and cleanup schedulers).
- **`api/v1/`**: Contains the route definitions, separated by logical domains:
  - `auth.py`, `users.py`, `admin.py`, `feedback.py`, `visits.py`
  - `tools/`: Contains route definitions for each specific tool (e.g., `qrcode`, `pdf`, `image`, `cad`, etc.).
- **`services/`**: Contains the core business logic, keeping routes clean:
  - `auth_service.py`, `user_service.py`, `email_service.py`, etc.
  - `tools/`: Contains the heavy lifting logic for individual tools (e.g., pdf manipulation, image background removal).
- **`models/`**: SQLModel database models representing tables like users, tools, and usage history.
- **`schemas/`**: Pydantic models for request/response validation (often overlapping or inheriting from DB models).
- **`core/`**: Core configuration classes (loading environment variables), custom exceptions, and security utilities.
- **`workers/`**: Contains `cleanup.py` which uses `BackgroundScheduler` to periodically sweep and delete old temporary files generated by tools.

## 🎨 Frontend Architecture

The frontend is a modern **Next.js** application using the **App Router** paradigm, providing both Server-Side Rendering (SSR) where appropriate and interactive Client-Side Components.

### Technology Stack
- **Framework**: Next.js 15 (React 18)
- **Styling**: Tailwind CSS with `tailwind-merge` and `clsx`
- **UI Components**: Radix UI primitives (accessible, unstyled components) often structured via a system like `shadcn/ui` based on the `src/components/ui/` folder presence.
- **Data Fetching & Caching**: `@tanstack/react-query` (React Query) combined with `axios` (`src/lib/api.ts`).
- **State Management**: Zustand and React Context (e.g., `AuthProvider`).
- **Icons**: `lucide-react`

### Project Structure (`frontend/src`)
- **`app/`**: Next.js App Router structure handling routing and page layouts.
  - `(auth)`: Route group for login/register pages.
  - `(marketing)`: Route group for landing pages and marketing content.
  - `dashboard/`: The main authenticated area, containing sub-routes for individual `/tools/...`.
  - `admin/`: Admin-specific dashboard and management pages.
- **`components/`**: Reusable React components.
  - `ui/`: Base UI building blocks (buttons, dialogs, inputs) wrappers around Radix UI.
  - `shared/`: Composite components used across different features (e.g., modals, specific cards).
  - `layout/`: Global layout components like Header, Sidebar, and Footer.
  - `tools/`: Components specific to tool interfaces.
- **`providers/`**: Context providers that wrap the application:
  - `AuthProvider`: Manages user authentication state, token refresh logic (using `js-cookie`), and login/logout functions. It also integrates with React Query to invalidate caches on auth changes.
- **`lib/`**: Utility functions and configuration:
  - `api.ts`: Pre-configured Axios instance for backend communication, potentially handling auth headers automatically.
  - `utils.ts`: General helper functions (e.g., class name merging).

## 🔄 Communication Flow
1. The user interacts with the Frontend UI.
2. The Frontend uses `axios` (often wrapped in React Query hooks for caching/loading states) to make asynchronous HTTP requests to the Backend API.
3. Access tokens and refresh tokens are securely stored in cookies (`js-cookie`). The Frontend sends the access token as a Bearer token or via cookies to the Backend.
4. The Backend (FastAPI) validates the request (Pydantic schemas), authenticates the user, processes the logic (via `services`), and interacts with the database (SQLModel) if necessary.
5. The Backend returns a JSON response to the Frontend, which then updates its UI state accordingly.
